1. **Adjust variable mutability for reconnect support:**
- The `_conn` (or `conn`) variable holding `MidiInputConnection` must be mutable: `let mut conn = midi_in.connect(...)?;` so it can be reassigned after reconnect.
- Store the connected device name from `select_midi_port()` as `connected_name: String` — this is the name used for liveness checks.
- Before the first `connect()` call, clone the sender for reconnection: `let reconnect_tx = tx.clone();` (or restructure so a sender is available for reconnect). After connecting, the original `tx` is consumed by `connect()` as the data parameter, so the clone is needed for creating new callbacks on reconnect.

2. **Add disconnect detection to the timeout branch** of the main loop (per D-09):
In the `Err(RecvTimeoutError::Timeout)` arm, after checking the `running` flag, add a port liveness check:
```rust
Err(std::sync::mpsc::RecvTimeoutError::Timeout) => {
    // Check if device is still connected
    if let Ok(probe) = midir::MidiInput::new("haydn-probe") {
        let ports = probe.ports();
        let still_connected = ports.iter().any(|p| {
            probe.port_name(p).ok().as_deref() == Some(connected_name.as_str())
        });
        if !still_connected {
            drop(conn); // Release the dead connection
            eprintln!("\n⚠ MIDI device disconnected. Waiting for device... (Ctrl+C to quit)");
            conn = wait_for_reconnect(
                &connected_name,
                reconnect_tx.clone(),
                &running,
            )?;
            eprintln!("✓ MIDI device reconnected. Resuming session.\n");
        }
    }
    continue;
}
```
**Critical:** The VM (`vm`), tuning engine (`engine`), and all computation state are NOT touched during disconnect/reconnect. Stack, memory, loop recordings — everything is preserved (per D-09, D-10).

3. **Implement `wait_for_reconnect()` function:**
```rust
fn wait_for_reconnect(
    device_name: &str,
    tx: std::sync::mpsc::Sender<MidiMsg>,
    running: &std::sync::Arc<std::sync::atomic::AtomicBool>,
) -> anyhow::Result<midir::MidiInputConnection<std::sync::mpsc::Sender<MidiMsg>>>
```
Implementation:
- Loop with `std::thread::sleep(Duration::from_secs(1))` intervals
- Each iteration: check `running.load(Ordering::Relaxed)`. If false, bail with `"Session interrupted while waiting for MIDI device"`
- Create `MidiInput::new("haydn-probe")`, call `.ports()`, search for a port whose `port_name()` matches `device_name`
- When found: create `MidiInput::new("haydn-reconnect")`, find the matching port again (probe was consumed by ports()), connect with `midi_callback` and `tx`, return the new `MidiInputConnection`
- Note: `MidiInput::new()` + `.ports()` is very cheap (~0.1ms) — fine for 1-second polling

4. **Enhance shutdown to print session summary** (per D-11):
After the main loop exits (when `running` becomes false OR channel disconnects), print a summary:
```rust
vm.close(); // Close any open loops cleanly
println!("\n─── Session Summary ───");
println!("Final stack ({} items): {:?}", vm.stack().len(), vm.stack());
if !vm.output().is_empty() {
    match std::str::from_utf8(vm.output()) {
        Ok(s) => println!("Program output: {}", s),
        Err(_) => println!("Program output (bytes): {:?}", vm.output()),
    }
}
println!("───────────────────────");
```

5. **Handle `RecvTimeoutError::Disconnected`** (all senders dropped):
This is an abnormal path — print a note and exit the loop. The normal exit path is Ctrl+C. Format: `eprintln!("Channel disconnected unexpectedly. Ending session.");` then break.
