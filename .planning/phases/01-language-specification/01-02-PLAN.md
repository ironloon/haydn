**Section 7: Turing Completeness**

Prove Haydn is Turing-complete by reduction from Brainfuck.

**7.1 Overview**: State that we prove TC by showing any Brainfuck program can be mechanically translated into a Haydn event sequence producing identical I/O and halting behavior.

**7.2 Brainfuck Reference**: List BF's 8 instructions (`>`, `<`, `+`, `-`, `.`, `,`, `[`, `]`) with their semantics. BF operates on a tape of cells (initialized to 0) with a movable pointer.

**7.3 Compilation Convention**: Define how BF's tape maps to Haydn:
- BF tape cell `i` → Haydn memory address `i`
- BF data pointer → tracked as a value on top of the Haydn stack
- Setup event: `Push(0)` initializes pointer at cell 0

**7.4 Instruction Mapping Table**: Map each BF instruction to a fixed Haydn event sequence. The pointer value is always maintained on the stack top throughout execution.

| BF | Haydn Events | Effect |
|----|-------------|--------|
| `>` | `Push(1), Op(add)` | Increment pointer |
| `<` | `Push(1), Op(sub)` | Decrement pointer |
| `+` | `Op(dup), Op(dup), Op(load), Push(1), Op(add), Op(rotate), Op(store)` | Load cell at pointer, increment by 1, store back. Stack trace: [ptr]→dup→[ptr ptr]→dup→[ptr ptr ptr]→load→[ptr ptr val]→Push(1)→[ptr ptr val 1]→add→[ptr ptr val+1]→rotate→[ptr val+1 ptr]→store→[ptr] with memory[ptr]=val+1 |
| `-` | `Op(dup), Op(dup), Op(load), Push(1), Op(sub), Op(rotate), Op(store)` | Same as `+` but subtract 1 |
| `.` | `Op(dup), Op(load), Op(print_char)` | Dup pointer (preserve it), load cell value, print as char |
| `,` | `Op(dup), Op(read), Op(swap), Op(store)` | Dup pointer, read input byte, swap so addr is on top for store. Stack trace: [ptr]→dup→[ptr ptr]→read→[ptr ptr input]→swap→[ptr input ptr]→store→[ptr] with memory[ptr]=input |
| `[` | `Op(dup), Op(load), Op(loop_start)` | Dup pointer, load cell value to stack top. loop_start peeks: if zero → skip body (while-loop semantics per spec Section 5); if nonzero → begin recording. |
| `]` | `Op(drop), Op(dup), Op(load), Op(loop_end)` | Drop stale check value, dup pointer, load current cell value, loop_end pops: nonzero → replay buffer; zero → exit loop. |

**IMPORTANT NOTE FOR EXECUTOR**: The `]` mapping must carefully handle the stack. After loop body execution, the stack state depends on what the body did. All BF instruction mappings preserve the pointer on stack top. The `Op(drop)` at `]` drops the previous cell-check value left on the stack by `Op(load)` from the `[` mapping (or from the prior iteration's `]` replay). Work through the BF program `[-]` (decrement cell 0 until zero) step-by-step to validate the mapping is correct. If the stack trace reveals issues, **adjust the mapping — correctness is paramount**. The table above is a starting point; the final mapping in the spec must be verified with at least one fully worked BF program trace.

**7.5 Proof Argument**: Three properties:
1. **Totality**: All 8 BF instructions have a defined mapping (shown in table)
2. **I/O Preservation**: BF `.` maps to `print_char`, BF `,` maps to `read` — identical byte I/O
3. **Halting Preservation**: BF `[`/`]` maps to `loop_start`/`loop_end` with while-loop semantics — loop entry/exit conditions are identical

**7.6 Conclusion**: Since Brainfuck is Turing-complete and every BF program mechanically compiles to a valid Haydn event sequence with identical I/O and halting behavior, Haydn is Turing-complete. QED.
