**Section 1: Introduction**
- Haydn is an esoteric programming language where live musical performance IS the program
- The VM receives events (from tuning files) and executes them on a stack machine
- This spec defines the VM semantics — musical input mapping is defined by tuning files (separate concern per D-04)

**Section 2: Computational Model**
- **Values**: Signed 64-bit integers (i64). Wrapping arithmetic on overflow/underflow.
- **Stack**: Unbounded LIFO stack of i64 values. Initially empty.
- **Memory**: Addressable array of i64 values, indexed by non-negative i64. Initially all zeros. Addresses are i64 values from the stack; negative addresses are invalid.
- **I/O**: Output stream (bytes); Input stream (bytes, 0 if unavailable). Non-blocking.
- **State**: The VM state is the tuple (stack, memory, loop_state, output_buffer, input_buffer).

**Section 4: Event Interface**
- The VM is event-driven. It receives a stream of `Event` values and processes them sequentially.
- Define the Event type:
  ```
  Event = Push(value: i64) | Op(opcode: Opcode)
  ```
- `Push(value)` pushes the value onto the stack. This is how notes become values (per D-01, D-03, D-05).
- `Op(opcode)` executes the named opcode per the Instruction Set table.
- `Push` is NOT an opcode — it is a distinct event type. The tuning file decides when to emit Push vs Op events.
- **Optional metadata**: Each event MAY carry metadata (velocity as u8 0-127, duration in milliseconds). v1 VM ignores metadata entirely. Metadata exists for forward compatibility with v2 features (D-12, D-14).
- **Event ordering**: Events are processed in the order received. The VM maintains no internal clock or timer.
- **Source independence**: The VM does not know or care whether events came from MIDI, microphone, or a test harness. All events are identical at the VM boundary.
- State: "The event interface is the ONLY way to drive the VM. There is no instruction pointer, no program counter, no stored program. The stream of events IS the program."

**Section 5: Control Flow — Loop Pedal Model**
Define the loop state machine with a complete transition table:

States: `Normal`, `Recording(depth)`, `Replaying(depth, position)`

**CRITICAL SEMANTIC CHOICE (per D-09 "direct Brainfuck reduction"):** `loop_start` PEEKS at the stack top. If zero (or stack empty), the VM skips to after the matching `loop_end` without recording or executing the body. If nonzero, the VM begins recording. This gives **while-loop semantics** (not do-while), enabling a direct 1:1 mapping from Brainfuck's `[` and `]`.

State machine transitions:

| Current State | Event | Action | Next State |
|---------------|-------|--------|------------|
| Normal | Push(v) | Push v onto stack | Normal |
| Normal | Op(loop_start) | Peek stack top. If zero or empty: skip to after matching loop_end. If nonzero: push new empty buffer onto loop stack. | Normal (if skipped) or Recording(1) |
| Normal | Op(loop_end) | No-op (unmatched) | Normal |
| Normal | Op(other) | Execute opcode | Normal |
| Recording(d) | Push(v) | Push v onto stack AND append Push(v) to current buffer | Recording(d) |
| Recording(d) | Op(loop_start) | Peek stack top. If zero/empty: append skip-marker to buffer, skip to matching loop_end within recording. If nonzero: push new buffer onto loop stack, append to parent buffer. | Recording(d) (if skipped) or Recording(d+1) |
| Recording(d) | Op(loop_end) at depth d=1 | Pop stack top. Nonzero: begin replaying current buffer from position 0. Zero: discard buffer, pop loop stack. | Replaying(1, 0) or Normal |
| Recording(d) | Op(loop_end) at depth d>1 | Pop stack top. Nonzero: begin replaying inner buffer. Zero: discard inner buffer, pop loop stack. | Replaying(d, 0) or Recording(d-1) |
| Recording(d) | Op(other) | Execute opcode AND append Op(other) to current buffer | Recording(d) |
| Replaying(d, pos) | (buffer[pos] = Push(v)) | Push v onto stack | Replaying(d, pos+1) |
| Replaying(d, pos) | (buffer[pos] = Op(loop_start)) | Peek stack top; if nonzero: push new buffer, begin recording nested loop. If zero: skip to matching loop_end in buffer. | Recording(d+1) or Replaying(d, pos_after_match) |
| Replaying(d, pos) | (buffer[pos] = Op(loop_end)) | Pop stack top. Nonzero: replay inner. Zero: exit inner. | Replaying or reduce depth |
| Replaying(d, pos) | (buffer[pos] = Op(other)) | Execute opcode | Replaying(d, pos+1) |
| Replaying(d, end_of_buffer) | — | Pop stack top. Nonzero: reset pos to 0 (replay again). Zero: discard buffer, pop loop stack. | Replaying(d, 0) or Recording(d-1) or Normal |

Key rules to state explicitly:
1. "During replay, live input events are QUEUED, not interleaved. The VM processes the entire replay buffer before accepting new live events." (Per research recommendation)
2. "Maximum loop nesting depth: 256. Exceeding this limit causes the loop_start to be a no-op."
3. "If the event stream ends while loops are open (Recording or Replaying state), all open loop buffers are discarded and the VM returns to Normal state."
4. "`loop_end` without a matching `loop_start` (in Normal state) is a no-op."
5. "During recording, opcodes are BOTH executed AND recorded. The loop body runs during its first recording pass, then conditionally replays."
6. "The stack top consumed by `loop_end` is popped (consumed), not peeked."
7. "`loop_start` PEEKS (does not consume) the stack top. If zero or empty, the entire loop body is skipped. This gives while-loop semantics enabling direct Brainfuck reduction (per D-09)."
8. "When loop_start skips (stack top is zero), it must find the matching loop_end accounting for nesting depth — inner loop_start/loop_end pairs are matched and skipped over."

**Section 6: Silence and Timing**
- "Silence is the absence of events. The VM is purely event-driven and has no internal clock. When no events arrive, no computation occurs."
- "The VM defines no timing semantics. Events are processed in order of receipt with no concept of tempo, rhythm, or duration."
- "Sustained/held notes: The spec assigns no special behavior to note duration. If the input layer sends events for sustained notes, the VM processes them like any other events. The tuning file decides what events (if any) to generate from sustained notes (D-13)."
- "Deferred to v2: Tempo-aware operations, dynamic-expression-based operations, rhythm-pattern detection (D-14)."
