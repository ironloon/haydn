<task type="auto">
  <name>Task 2: Create tag-triggered release workflow</name>
  <files>.github/workflows/release.yml</files>
  <action>
    Per D-07: Create `.github/workflows/release.yml` implementing the draft approval model.

    **Trigger:** Push of tags matching `v*` (e.g., `v0.0.1`).

    **Job 1: `test`** — Run full test matrix (same 3-OS matrix as CI) to validate the tagged commit. Must pass before release creation.

    **Job 2: `draft-release`** — Needs `test` job. Uses `softprops/action-gh-release@v2` (or equivalent) to create a GitHub Release in **draft** state. Tag name = release name. Auto-generate release notes from commits.

    **Job 3: `publish`** — Triggers on `release` event with type `published` (user manually clicks "Publish release" on the draft). Per D-08, publishes crates in strict dependency order with a delay between each for crates.io index propagation:

    ```yaml
    steps:
      - uses: actions/checkout@v4
      - uses: dtolnay/rust-toolchain@stable
      - run: cargo publish -p haydn-vm
        env:
          CARGO_REGISTRY_TOKEN: ${{ secrets.CARGO_REGISTRY_TOKEN }}
      - run: sleep 30  # crates.io index propagation
      - run: cargo publish -p haydn-tuning
        env:
          CARGO_REGISTRY_TOKEN: ${{ secrets.CARGO_REGISTRY_TOKEN }}
      - run: sleep 30
      - run: cargo publish -p haydn-audio
        env:
          CARGO_REGISTRY_TOKEN: ${{ secrets.CARGO_REGISTRY_TOKEN }}
      - run: sleep 30
      - run: cargo publish -p haydn
        env:
          CARGO_REGISTRY_TOKEN: ${{ secrets.CARGO_REGISTRY_TOKEN }}
      - run: sleep 30
      - run: cargo publish -p haydn-performer
        env:
          CARGO_REGISTRY_TOKEN: ${{ secrets.CARGO_REGISTRY_TOKEN }}
    ```

    Per D-09: Token comes from `secrets.CARGO_REGISTRY_TOKEN` GitHub repo secret (user must configure this — noted but not a checkpoint since it's a one-click GitHub UI action documented in the release checklist).

    Note: The `publish` job must be in the SAME workflow file but with a separate trigger (`on: release: types: [published]`). Since GitHub Actions allows multiple `on` triggers, structure the file with both tag push and release event triggers, using `if` conditions on jobs to route correctly.

    Actually, for clarity, it's better to have the publish job in the same file using the release trigger. The workflow file can have:
    ```yaml
    on:
      push:
        tags: ['v*']
      release:
        types: [published]
    ```
    Then:
    - `test` job: runs on tag push only (`if: github.event_name == 'push'`)
    - `draft-release` job: runs on tag push, needs test (`if: github.event_name == 'push'`)
    - `publish` job: runs on release published only (`if: github.event_name == 'release'`)
  </action>
