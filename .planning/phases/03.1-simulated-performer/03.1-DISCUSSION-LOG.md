## Reverse Mapping Strategy

| Option | Description | Selected |
|--------|-------------|----------|
| Build reverse lookup at compile time | HashMap<Event, u8> from tuning rules | |
| Preferred note annotations | `[reverse]` section in TOML tuning file | |
| Algorithmic "most musical" selection | Minimize melodic leaps, preferred octave | |
| **Eliminated entirely** | Simulator plays notes forward — no reverse needed | ✓ |

**User's choice:** The user reframed the simulator's architecture: it's a digital musician that plays music and Haydn listens. The simulator doesn't need to go event→note because it plays LilyPond notes directly. The tuning engine's existing note→event direction handles computation when wired up later.

**Notes:** This was a fundamental architecture insight from the user that simplified the entire phase. The simulator has zero awareness of Haydn.

## Audio Synthesis & Playback

| Option | Description | Selected |
|--------|-------------|----------|
| Simple sine wave | Pure, functional, minimal | |
| Piano-like (sampled) | Realistic but heavy | |
| Basic synth (sine + envelope) | Good middle ground | |
| Layered architecture (configurable) | Multiple layers, dial fidelity up/down | ✓ |

**User's choice:** Layered synthesis with full richness — "go ham on the simulator." User specifically mentioned vibrato, realistic sounds, and leveraging open-source MIDI instruments and SoundFont libraries. The insight: synthesis fidelity affects what you hear, which affects tuning design decisions.

**Layers defined:**
- Layer 0: Pure sine
- Layer 1: Shaped sine + ADSR envelope (default)
- Layer 2: Rich waveform (harmonics, blending)
- Layer 3: Expressive (vibrato, tremolo, dynamics, legato)
- Layer 4: SoundFont/SF2 playback (open-source instruments)
- Layer 5: MIDI output to external synth/DAW

**Notes:** User emphasized "configurable in layers" because different synthesis fidelities may change test results and influence Haydn's direction. Also: explore `rustysynth`, `oxisynth` for SF2 in Rust.

## Output & VM Display

| Option | Description | Selected |
|--------|-------------|----------|
| Real-time interleaved (verbose) | Per-note text with tuning match + VM state | |

| Side-by-side TUI | Split pane: score + VM state | |
| Configurable verbosity (all modes) | `--display quiet\|verbose\|tui` | |
| **Best music player UX (standalone)** | Music player display, no VM awareness | ✓ |

**User's choice:** "Forget integrations with Haydn for this phase, focus on best user experience for the music playback sim, however definitely consider interoperability." The simulator shows music playback UX only — current note, scrolling score, progress. No stack state, no events, no tuning terminology. `--quiet` for headless use.

**Notes:** This decision reinforces D-04 (standalone crate, zero Haydn deps). The display is a music player, not a debugger.
