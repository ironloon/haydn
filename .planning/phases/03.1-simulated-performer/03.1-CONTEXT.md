# Phase 3.1: Simulated Performer - Context

**Gathered:** 2026-03-29
**Status:** Ready for planning

<domain>
## Phase Boundary

Standalone music playback tool that reads LilyPond notation, synthesizes audio through configurable layered synthesis, and plays it through the computer's speakers. The simulator is a **digital musician** — it plays music, period. It has no awareness of the Haydn VM, tuning engine, or event pipeline. However, it is architecturally designed for interoperability: it exposes a public note event stream API and uses MIDI note numbers as the interchange format, so downstream phases can wire it into Haydn's input pipeline.

The primary purpose is enabling musicality evaluation of tuning files — the user hears what common instruction sequences sound like on a given tuning before real MIDI hardware or microphone input exists.

</domain>

<decisions>
## Implementation Decisions

### Input Format
- **D-01:** LilyPond notation (subset). The simulator reads `.ly` files containing LilyPond note syntax. A custom subset parser converts notation to MIDI note numbers — we do NOT need the full LilyPond engraving engine.
- **D-02:** Supported LilyPond subset: note names (`c d e f g a b`), octave marks (`'` up, `,` down), accidentals (`is`/`es` for sharp/flat), durations (digits: `4` = quarter, `8` = eighth, etc.), rests (`r`), barlines (`|`), and absolute/relative mode. Enough to express any melodic sequence.
- **D-03:** Thematic fit is intentional — LilyPond is the notation system of serious classical music, the cultural peer of the project's namesake (Joseph Haydn). ABC notation and MML were considered and rejected for this reason.

### Architecture & Interoperability
- **D-04:** The simulator is a standalone crate (e.g., `haydn-sim` or `haydn-performer`) with zero dependencies on `haydn-vm` or `haydn-tuning`. Clean crate boundary.
- **D-05:** Exposes a public note event stream API — a sequence of `(midi_note: u8, duration: Duration, velocity: u8)` events (or similar) that downstream consumers can tap into. This is the interoperability surface for Phase 4+.
- **D-06:** MIDI note numbers are the interchange format between the simulator and the tuning engine. The simulator produces MIDI note numbers; a future integration layer feeds those into `TuningEngine::map_note()`.
- **D-07:** No reverse mapping needed. The simulator plays notes forward (LilyPond → MIDI notes → audio). It never needs to go event→note. The tuning engine's existing note→event direction handles computation when wired up later.

### Audio Synthesis (Layered)
- **D-08:** Layered synthesis architecture. Multiple synthesis backends, swappable via CLI flag. The synthesis engine is trait-based/modular so layers are independent.
- **D-09:** Layer 0 — Pure sine wave. Built-in, zero external deps beyond audio output. Functional baseline.
- **D-10:** Layer 1 — Shaped sine with ADSR envelope (attack/decay/sustain/release). Notes feel like notes, not beeps. Default layer.
- **D-11:** Layer 2 — Rich waveform synthesis. Saw/triangle/square blending, harmonics. More instrument-like character.
- **D-12:** Layer 3 — Expressive. Vibrato, tremolo, velocity dynamics, legato transitions. Can pull expression data from LilyPond annotations (dynamics, articulations) when present; applies automatic expression based on note duration/pitch when not.
- **D-13:** Layer 4 — SoundFont (SF2) playback. Load open-source SoundFont files (FluidR3, Salamander Grand Piano, etc.) for realistic instrument sounds. Investigate Rust SF2 libraries (`rustysynth`, `oxisynth`).
- **D-14:** Layer 5 — MIDI output. Instead of synthesizing, send MIDI to an external synth/DAW via virtual MIDI port or write a `.mid` file. Maximum fidelity with zero synthesis work. The user's own instruments handle sound.
- **D-15:** Go rich — leverage open-source MIDI instruments, SoundFont libraries, and the broader ecosystem. The fidelity of the simulation affects what the user hears, which affects tuning design decisions.
- **D-16:** CLI flags: `--synth builtin|soundfont|midi` for synthesis backend, `--fidelity 0-4` for expression level within a backend, `--soundfont path/to/file.sf2` for SF2 loading, configurable tempo via `--bpm` or derived from LilyPond durations.

### Display & User Experience
- **D-17:** Best music player UX — the simulator shows what a music player should show. Current note, scrolling score view, progress indicator. No VM state, no events, no tuning terminology.
- **D-18:** Default display: real-time note indicator with progress bar and measure tracking. Clean, musical presentation.
- **D-19:** `--quiet` flag for headless/scripting use — just play audio, exit.
- **D-20:** This is NOT a Phase 5 TUI. No split-pane VM observation. The display is focused entirely on the music playback experience.

### Agent's Discretion
- Specific crate name (`haydn-sim` vs `haydn-performer` vs other)
- Audio output library choice (`rodio`, `cpal`, or other) — research during planning
- SoundFont library choice (`rustysynth`, `oxisynth`, or other) — research during planning
- MIDI output library choice for virtual MIDI ports
- Exact LilyPond parser implementation approach (hand-written vs parser combinator like `nom`)
- Internal trait design for the layered synthesis engine
- Default tempo when LilyPond score doesn't specify one
- Which open-source SoundFonts to bundle or recommend

</decisions>

<code_context>
## Existing Code Insights

### Reusable Assets
- `crates/haydn-tuning/src/engine.rs`: `TuningEngine::map_note(u8) -> Option<Event>` — future integration point. Simulator doesn't use this directly but its output format (MIDI note numbers) is designed to feed it.
- `crates/haydn-tuning/tunings/piano.toml`: Default tuning with specific MIDI note assignments. Use this to author example LilyPond scores that exercise common instruction sequences.
- Cargo workspace pattern: root `Cargo.toml` with `crates/` members. New simulator crate follows the same structure.

### Established Patterns
- Cargo workspace with crate-per-concern: `haydn-vm` (computation), `haydn-tuning` (mapping), `haydn` (binary). Simulator becomes another workspace member.
- Zero-dep philosophy in core crates. Simulator will necessarily introduce audio deps — keep them isolated to the sim crate.

### Integration Points
- The simulator is a new workspace member alongside existing crates. No code changes to existing crates needed.
- Future integration (Phase 4+): a thin adapter feeds the simulator's note event stream into `TuningEngine::map_note()` → `HaydnVm::feed()`. The simulator itself doesn't change.

</code_context>

<specifics>
## Specific Ideas

- The choice of LilyPond is deliberately thematic — classical music notation for a project named after a classical composer. This reinforces the project's identity and signals to musicians that the tool speaks their language.
- Synthesis fidelity matters for tuning evaluation: a sine wave Hello World might sound fine, but realistic piano timbre with vibrato could reveal that certain note sequences create ugly beating or awkward phrasing. The layered architecture lets the user dial fidelity up/down to test this.
- Example `.ly` scores should exercise common Haydn instruction patterns (push-push-op sequences from Hello World, loop structures from FizzBuzz) using the piano tuning's note assignments, so the user can hear whether "programming sounds musical."
- MIDI output mode (Layer 5) enables the user to pipe simulated performances into any DAW or external instrument — GarageBand, Logic, a hardware synth — for maximum realism with zero synthesis effort.

</specifics>

<deferred>
## Deferred Ideas

- **VM integration display** — Showing stack state, events, and tuning matches alongside music playback. Belongs in Phase 4 (MIDI pipeline) or Phase 5 (CLI visualization) when the full pipeline exists.
- **Live LilyPond editing** — Watch a `.ly` file for changes and re-play automatically. Hot-reload for tuning iteration. Could be a future enhancement.
- **Score generation from BF** — Auto-generate a LilyPond score from a BF program + tuning file (reverse mapping). Useful but separate from the playback tool.
- **Multi-voice/polyphonic playback** — LilyPond supports multiple voices. The subset parser only handles monophonic for v1. Expand later if needed.
- **Audio file export** — `--output wav` or `--output mp3` to save the performance as an audio file. Useful for sharing but not core to evaluation.

</deferred>
